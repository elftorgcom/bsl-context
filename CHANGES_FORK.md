# Изменения в форке bsl-context

В данном файле описаны изменения, внесенные в форк [elftorgcom/bsl-context](https://github.com/elftorgcom/bsl-context) относительно оригинального репозитория [Regsorm/bsl-context](https://github.com/Regsorm/bsl-context).

---

## Краткое описание: Что сделано и зачем

Основная цель изменений — улучшить удобство развертывания и использования MCP-сервера (Model Context Protocol) для контекста платформы 1С:Предприятие в веб-окружении (например, на хостинге Alwaysdata) и локально.

**Ключевые доработки:**
1. **Интерактивная стартовая веб-страница (`/`)**: Добавлен красивый веб-интерфейс, который отображает текущий статус сервера (работает ли он, версия, загружен ли индекс 1С, сколько сущностей проиндексировано) и предоставляет готовые конфигурационные файлы для подключения к клиентам (Antigravity, Codex, Claude Desktop) с возможностью копирования в один клик.
2. **Улучшенная сетевая совместимость**:
   * Реализована встроенная поддержка IPv6 адресов (например, `::1` или `::`) в конфигурации без необходимости оборачивать их в квадратные скобки.
   * Добавлена возможность полного отключения валидации заголовков `Host` (через маску `*` или пустой список разрешенных хостов), что решает проблему с ошибками `403 Forbidden: Host header is not allowed` у стандартных HTTP-клиентов.
   * Добавлена опция `external_domain` для корректного формирования ссылок подключения на стартовой странице при работе за прокси или на внешнем хостинге.
3. **Обновление версий и CI/CD**:
   * Версия проекта повышена с `0.4.0` до `0.4.3`.
   * Исправлены ошибки сборки, связанные с расхождением версий в `Cargo.lock`.
   * CI-скрипты GitHub Actions адаптированы для предотвращения предупреждений об устаревании Node.js 20.

## Подробный реестр изменений по компонентам

### 1. Сетевое взаимодействие и конфигурация сервера

#### [MODIFY] [crates/server/src/main.rs](file:///C:/AI/bsl-context/bsl-context-main/crates/server/src/main.rs)
* **Что изменено**: Переработан механизм парсинга адреса прослушивания (IP и порт).
* **Зачем**: Раньше строка адреса формировалась простым форматированием `format!("{}:{}", cfg.host, cfg.port)` и затем парсилась в `SocketAddr`. Это приводило к ошибкам при указании IPv6-адресов (например, `::1`), так как сетевой стек Rust требует для них квадратных скобок в строковом представлении (например, `[::1]:8007`).
* **Как теперь**: Сначала значение `cfg.host` парсится отдельно в тип `std::net::IpAddr`, после чего создается структура `SocketAddr::new(ip, cfg.port)`. Это нативно поддерживает как IPv4, так и IPv6 без дополнительных символов форматирования.

#### [MODIFY] [crates/server/src/config.rs](file:///C:/AI/bsl-context/bsl-context-main/crates/server/src/config.rs)
* **Что изменено**: В структуру конфигурации `Config` добавлено новое необязательное поле `external_domain: Option<String>` (по умолчанию `None`).
* **Зачем**: Позволяет явно задать внешний домен сервера (например, `bsl-context.alwaysdata.net`), используемый стартовой страницей для генерации параметров подключения к клиентам.

#### [MODIFY] [configs/config.toml.example](file:///C:/AI/bsl-context/bsl-context-main/configs/config.toml.example)
* **Что изменено**: В файл примера конфигурации добавлено описание и закомментированная строка нового параметра `# external_domain = "bsl-context.alwaysdata.net"`.

## Как обновлять форк при изменениях в оригинальном репозитории (upstream)

Чтобы ваши изменения не пропали и корректно наложились на обновления из основного проекта, рекомендуется использовать один из следующих подходов.

### Вариант A: Перебазирование (Git Rebase) — Рекомендуемый
Этот способ переносит ваши коммиты так, будто они были сделаны поверх самой последней версии оригинального проекта. Это сохраняет историю чистой и линейной.

1. **Подключите оригинальный репозиторий как `upstream`** (если еще не сделано):
   ```bash
   git remote add upstream https://github.com/Regsorm/bsl-context.git
   ```
2. **Получите последние изменения из upstream**:
   ```bash
   git fetch upstream
   ```
3. **Перейдите на свою основную ветку**:
   ```bash
   git checkout main
   ```
4. **Выполните rebase**:
   ```bash
   git rebase upstream/main
   ```
   *Если возникнут конфликты в коде (например, в файлах, которые меняли и вы, и автор оригинала):*
   * Git приостановит процесс rebase.
   * Откройте конфликтующие файлы, разрешите конфликты (оставьте нужные участки кода).
   * Зафиксируйте разрешение: `git add <имя_файла>`
   * Продолжите процесс: `git rebase --continue` (не делайте коммит вручную!).
5. **Отправьте изменения в свой GitHub-репозиторий**:
   Поскольку rebase переписывает историю коммитов, обычный `git push` не сработает. Понадобится принудительная отправка:
   ```bash
   git push origin main --force-with-lease
   ```

---

### Вариант Б: Слияние (Git Merge)
Менее требовательный к истории способ, который просто создает коммит слияния. Не требует force-push.

1. **Получите изменения**:
   ```bash
   git fetch upstream
   ```
2. **Выполните слияние**:
   ```bash
   git checkout main
   git merge upstream/main
   ```
3. **Разрешите конфликты** (если они возникнут), сделайте коммит слияния и отправьте изменения:
   ```bash
   git push origin main
   ```

---

### Вариант В: Использование патч-файла (Резервный вариант)
Если вы хотите полностью сбросить ветку форка до состояния оригинала, но сохранить свои наработки в виде отдельного файла, чтобы применить их позже.

1. **Создайте файл патча** со всеми вашими изменениями относительно upstream:
   ```bash
   git diff upstream/main..main > my_changes.patch
   ```
   *Файл `my_changes.patch` запишется локально. В нем будут содержаться все ваши диффы.*
2. **Сбросьте ветку `main`** до точной копии upstream:
   ```bash
   git checkout main
   git reset --hard upstream/main
   ```
3. **Примените патч обратно**:
   ```bash
   git apply my_changes.patch
   ```
4. **Закоммитьте изменения и сделайте force-push**:
   ```bash
   git add .
   git commit -m "Apply fork modifications"
   git push origin main --force
   ```
